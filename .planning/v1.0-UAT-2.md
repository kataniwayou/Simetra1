## Tests

### 1. Same metric distinguished by labels across devices
expected: Base labels (site, device_name, device_ip, device_type) on every measurement ensure same metric name on different devices produces distinct OTLP time series
result: PASS

### 2. Same metric distinguished by dynamic labels (OidRole.Label)
expected: OID entries with Role=Label produce dynamic tags appended to the base labels, enabling further per-metric differentiation (e.g., port_number label on trap definitions)
result: ISSUE (severity: medium)
notes: |
  NPB state polls lacked port_number Label OID. Traps had it but polls did not.
  Also surfaced need for port-indexed OIDs and unique MetricName per port for
  multi-port monitoring (MetricName = registry key, PropertyName = OTLP instrument).
fix: Added PortLogicalPortNumberOid as OidRole.Label to all 3 NPB state polls.

### 3. .NET runtime metrics enabled for leader
expected: AddRuntimeInstrumentation() is called in OTLP setup, enabling GC/thread pool/process metrics; leader exports them via RoleGatedExporter
result: ISSUE (severity: high)
notes: |
  Runtime metrics ARE collected (AddRuntimeInstrumentation present) but
  RoleGatedExporter<Metric> drops ALL metric batches when follower —
  including System.Runtime metrics that should export regardless of role.
fix: Created MetricRoleGatedExporter that filters by MeterName — gates only
  Simetra.Metrics, lets System.Runtime through on followers.

### 4. .NET runtime metrics collected but NOT exported by follower
expected: Follower should export runtime metrics (System.Runtime) while dropping application metrics (Simetra.Metrics)
result: ISSUE (same as Test 3)
notes: Same root cause — single RoleGatedExporter gates everything.
fix: Same fix as Test 3.

### 5. Port distinguishing - duplicate MetricName overwrites in registry
expected: If user adds two MetricPolls with same MetricName but different port OIDs to the same device, second overwrites first in PollDefinitionRegistry
result: ISSUE (severity: medium)
notes: |
  Surfaced that modules carried DeviceName/IpAddress as device identity
  but should be type-level only. Module defines type behavior (OIDs, traps);
  config defines device identity (name, IP). Having both creates confusion
  about which is the source of truth.
fix: Removed DeviceName/IpAddress from IDeviceModule interface. Refactored
  DeviceRegistry, PollDefinitionRegistry, DeviceChannelManager, and scheduling
  to use DeviceType matching against config devices.

### 6. Validation failure logging visibility
expected: If startup validation fails and Console=false, OTLP unreachable, the error should still be visible
result: ISSUE (severity: low)
notes: OptionsValidationException could be invisible in headless deployments.
fix: Added try/catch in Program.cs that writes validation failures to Console.Error.

### 7. Port distinguishing - NPB trap definitions include port label
expected: NPB trap definitions (port_link_up, port_link_down) include port_number as OidRole.Label
result: PASS (already correct before fixes)
