    **Complex nested options (validated via IValidateOptions):**

    10. **DevicesOptions.cs** (Section: "Devices"):
        This is a WRAPPER class. The JSON "Devices" key is an array, but IOptions needs a class.
        Use the pattern: bind the parent section and have a `List<DeviceOptions>` property.
        ACTUALLY -- "Devices" is a top-level array in the JSON. Use `configuration.GetSection("Devices")` to bind.
        The simplest approach: Create DevicesOptions as a wrapper with a single property:
        ```csharp
        public class DevicesOptions
        {
            public const string SectionName = "Devices";
            public List<DeviceOptions> Devices { get; set; } = [];
        }
        ```
        BUT -- binding a JSON array to a wrapper class requires the JSON to be `{"Devices": {"Devices": [...]}}` which is wrong. Instead, since the "Devices" key at root IS an array, we need a different approach.

        CORRECT APPROACH: Use a custom binding. The simplest .NET pattern for binding a root-level array is to use `IConfigureOptions<DevicesOptions>`:
        ```csharp
        public class DevicesOptions
        {
            public const string SectionName = "Devices";
            public List<DeviceOptions> Devices { get; set; } = [];
        }
        ```
        And in registration, bind it manually:
        ```csharp
        services.AddOptions<DevicesOptions>()
            .Configure<IConfiguration>((options, config) =>
            {
                config.GetSection(DevicesOptions.SectionName)
                    .Bind(options.Devices);
            })
            .ValidateOnStart();
        ```
        This binds the JSON array at "Devices" directly into the `Devices` list property.

    **Part A: Create IValidateOptions validators**

    All validators go in `namespace Simetra.Configuration.Validators;` and implement `IValidateOptions<T>`.

    1. **SiteOptionsValidator.cs** -- IValidateOptions<SiteOptions>:
       - Validate Name is not null/whitespace (required)
       - Return descriptive error: "Site:Name is required"

    2. **LeaseOptionsValidator.cs** -- IValidateOptions<LeaseOptions>:
       - Validate Name not null/whitespace
       - Validate Namespace not null/whitespace
       - Validate DurationSeconds > RenewIntervalSeconds (lease must outlive renew interval)
       - Return descriptive errors with full path: "Lease:Name is required", "Lease:DurationSeconds must be greater than Lease:RenewIntervalSeconds"

    3. **SnmpListenerOptionsValidator.cs** -- IValidateOptions<SnmpListenerOptions>:
       - Validate BindAddress not null/whitespace
       - Validate CommunityString not null/whitespace
       - Validate Version is exactly "v2c" (only supported version)
       - Validate Port in range 1-65535
       - Return descriptive errors: "SnmpListener:Version must be 'v2c'"

    4. **DevicesOptionsValidator.cs** -- IValidateOptions<DevicesOptions>:
       This is the CRITICAL validator. ValidateDataAnnotations does NOT validate nested objects.
       Must manually walk the entire object graph:
       - Empty Devices[] is VALID (per CONTEXT.md)
       - For each device[i]:
         - Name is required: "Devices[{i}].Name is required"
         - IpAddress is required: "Devices[{i}].IpAddress is required"
         - DeviceType is required: "Devices[{i}].DeviceType is required"
         - DeviceType must be a known type. For Phase 1, maintain a static HashSet of known device types: {"router", "switch", "loadbalancer", "simetra"} (case-insensitive comparison). Unknown type: "Devices[{i}].DeviceType '{type}' is not a registered device type"
       - For each device[i].MetricPolls[j]:
         - MetricName is required
         - IntervalSeconds must be > 0
         - Oids must not be empty (at least one OID per poll)
       - For each device[i].MetricPolls[j].Oids[k]:
         - Oid is required
         - PropertyName is required
         - Role must be a valid OidRole enum value

    5. **OtlpOptionsValidator.cs** -- IValidateOptions<OtlpOptions>:
       - Validate Endpoint not null/whitespace
       - Validate ServiceName not null/whitespace

    Note: HeartbeatJobOptions, CorrelationJobOptions, LivenessOptions, ChannelsOptions, and LoggingOptions have simple enough validation that DataAnnotations [Range]/[Required] suffice. They do NOT need IValidateOptions implementations.

       services.AddSingleton<IValidateOptions<SnmpListenerOptions>, SnmpListenerOptionsValidator>();

       services.AddSingleton<IValidateOptions<OtlpOptions>, OtlpOptionsValidator>();

       });
