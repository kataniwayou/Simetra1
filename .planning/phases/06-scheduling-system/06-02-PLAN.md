    **Execute method:**
    ```csharp
    public async Task Execute(IJobExecutionContext context)
    {
        // SCHED-08: Read correlationId BEFORE execution
        var correlationId = _correlation.CurrentCorrelationId;
        var jobKey = context.JobDetail.Key.ToString();

        try
        {
            // Read device name and metric name from JobDataMap
            var deviceName = context.MergedJobDataMap.GetString("deviceName")!;
            var metricName = context.MergedJobDataMap.GetString("metricName")!;

            // Look up DeviceInfo by name
            if (!_deviceRegistry.TryGetDeviceByName(deviceName, out var device))
            {
                _logger.LogWarning(
                    "State poll job {JobKey}: device {DeviceName} not found in registry",
                    jobKey, deviceName);
                return;
            }

            // Look up PollDefinitionDto by (deviceName, metricName)
            if (!_pollRegistry.TryGetDefinition(deviceName, metricName, out var definition))
            {
                _logger.LogWarning(
                    "State poll job {JobKey}: definition {MetricName} not found for device {DeviceName}",
                    jobKey, metricName, deviceName);
                return;
            }

            // Build SNMP GET request variables from definition OIDs
            var variables = definition.Oids
                .Select(o => new Variable(new ObjectIdentifier(o.Oid)))
                .ToList();

            // SNMP GET with 5000ms timeout
            var endpoint = new IPEndPoint(IPAddress.Parse(device.IpAddress), 161);
            var community = new OctetString(_listenerOptions.CommunityString);

            IList<Variable> response = await Messenger.GetAsync(
                VersionCode.V2,
                endpoint,
                community,
                variables,
                context.CancellationToken);

            // Extract + Process (Layer 3/4 -- bypass Layer 2 channels per PIPE-06)
            var result = _extractor.Extract(response, definition);
            _coordinator.Process(result, device, correlationId);

            _logger.LogDebug(
                "State poll completed for {DeviceName}/{MetricName} [CorrelationId: {CorrelationId}]",
                deviceName, metricName, correlationId);
        }
        catch (OperationCanceledException)
        {
            // Shutdown signal -- let it propagate (do not log as error)
            throw;
        }
        catch (Exception ex)
        {
            _logger.LogError(ex,
                "State poll job {JobKey} failed [CorrelationId: {CorrelationId}]",
                jobKey, correlationId);
        }
        finally
        {
            // SCHED-08: Stamp liveness vector on completion (always, even on failure)
            _liveness.Stamp(jobKey);
        }
    }
    ```

    **Required usings:**
    ```csharp
    using System.Net;
    using Lextm.SharpSnmpLib;
    using Lextm.SharpSnmpLib.Messaging;
    using Microsoft.Extensions.Logging;
    using Microsoft.Extensions.Options;
    using Quartz;
    using Simetra.Configuration;
    using Simetra.Pipeline;
    ```

    **Important details:**
    - The SNMP GET target port is 161 (standard SNMP agent port), NOT the listener port (162). The listener port is for receiving traps.
    - Community string comes from `SnmpListenerOptions.CommunityString` (shared for all SNMP operations in v1).
    - Timeout: `Messenger.GetAsync` uses the CancellationToken from Quartz context. For explicit timeout, if SharpSnmpLib does not support CancellationToken-based timeout natively, you can wrap with a `Task.WhenAny` or use the overload that accepts a timeout int parameter. Check the API -- `Messenger.GetAsync` has an overload with `int timeout` parameter in milliseconds. Use 5000ms. The signature is:
      `GetAsync(VersionCode version, IPEndPoint endpoint, OctetString community, IList<Variable> variables, int timeout)` -- use this overload with timeout=5000 if the CancellationToken overload is not available. If both exist, prefer the CancellationToken version. Check 06-RESEARCH.md for the correct signature.
    - OperationCanceledException should be rethrown (not caught) to respect shutdown signals.
    - The `return` statements in the "not found" branches skip the SNMP poll but still execute the finally block (liveness stamp), which is correct -- the job completed (with no work), so it's "alive."

    **MetricPollJob is structurally identical to StatePollJob** with the following differences:
    - Class name: `MetricPollJob` (not `StatePollJob`)
    - Logger type: `ILogger<MetricPollJob>`
    - Log messages reference "Metric poll" instead of "State poll"
    - The job is for Source=Configuration polls, but the Execute logic is exactly the same -- the Source routing (Branch A only vs Branch A + Branch B) is handled by IProcessingCoordinator based on the PollDefinitionDto.Source field, NOT by the job itself. The job just polls, extracts, and processes.

    **Class structure:**
    Identical constructor injection as StatePollJob:
    - `IDeviceRegistry`, `IPollDefinitionRegistry`, `ISnmpExtractor`, `IProcessingCoordinator`, `ICorrelationService`, `ILivenessVectorService`, `IOptions<SnmpListenerOptions>`, `ILogger<MetricPollJob>`

    **Execute method:**
    Identical flow as StatePollJob:
    1. Read correlationId before execution
    2. Get deviceName and metricName from JobDataMap
    3. Look up DeviceInfo by name
    4. Look up PollDefinitionDto by (deviceName, metricName)
    5. Build Variable list from definition OIDs
    6. SNMP GET to device:161 with community string
    7. Extract response via ISnmpExtractor
    8. Process via IProcessingCoordinator
    9. Stamp liveness in finally block

    **Required usings:** Same as StatePollJob.

    **Why two separate classes instead of one generic PollJob?**
    Even though the logic is identical, SCHED-02 and SCHED-03 specify separate state and metric poll job types. More importantly, `[DisallowConcurrentExecution]` scopes per JobKey (not per class), so a single class would work. However, two classes provide:
    - Clear semantic separation in logs and scheduler diagnostics
    - Different JobKey prefixes ("state-poll-" vs "metric-poll-")
    - Future extensibility if state polls need different behavior (e.g., retries, different timeout)
    - Follows the established naming in the research doc

    **Implementation note:** The code is nearly copy-paste from StatePollJob with log message differences. This duplication is acceptable for two small classes (~60 lines each). Extracting a shared base class would add complexity without value at this stage.
