    **IJobIntervalRegistry.cs** (namespace Simetra.Pipeline):
    - Interface with two methods:
      - `void Register(string jobKey, int intervalSeconds)` -- called during AddScheduling for each job
      - `bool TryGetInterval(string jobKey, out int intervalSeconds)` -- called by LivenessHealthCheck

    **JobIntervalRegistry.cs** (namespace Simetra.Pipeline):
    - Sealed class implementing IJobIntervalRegistry
    - Backed by `Dictionary<string, int>(StringComparer.Ordinal)`
    - Register sets/overwrites; TryGetInterval delegates to dictionary TryGetValue

    **StartupHealthCheck.cs** (namespace Simetra.HealthChecks):
    - Sealed class implementing IHealthCheck
    - Inject ICorrelationService via constructor
    - CheckHealthAsync: return Healthy if `!string.IsNullOrEmpty(_correlation.CurrentCorrelationId)`, otherwise Unhealthy("First correlationId not yet generated")
    - HLTH-03: Pipeline is wired (services resolved) and first correlationId exists

    **ReadinessHealthCheck.cs** (namespace Simetra.HealthChecks):
    - Sealed class implementing IHealthCheck
    - Inject IDeviceChannelManager and ISchedulerFactory via constructor
    - CheckHealthAsync:
      1. Check `_channels.DeviceNames.Count > 0` -- if zero, return Unhealthy("No device channels registered")
      2. Resolve scheduler via `await _schedulerFactory.GetScheduler(ct)`, check `scheduler.IsStarted && !scheduler.IsShutdown` -- if not, return Unhealthy("Quartz scheduler is not running")
      3. Return Healthy()
    - HLTH-04: All device channels open + Quartz scheduler running

    **LivenessHealthCheck.cs** (namespace Simetra.HealthChecks):
    - Sealed class implementing IHealthCheck
    - Inject ILivenessVectorService, IJobIntervalRegistry, IOptions&lt;LivenessOptions&gt;, ILogger&lt;LivenessHealthCheck&gt;
    - Store `_graceMultiplier` from options in constructor
    - CheckHealthAsync:
      1. Get all stamps via `_liveness.GetAllStamps()`
      2. For each stamp, look up interval via `_intervals.TryGetInterval(jobKey, out var intervalSeconds)` -- skip unknown jobs
      3. Compute threshold = `TimeSpan.FromSeconds(intervalSeconds * _graceMultiplier)`
      4. Compute age = `DateTimeOffset.UtcNow - lastStamp`
      5. If age > threshold, add to staleEntries dictionary with {ageSeconds, thresholdSeconds, lastStamp}
      6. If staleEntries.Count > 0: log Warning "Liveness check failed: {StaleCount} stale job(s) detected" and return Unhealthy with data dictionary (HLTH-05)
      7. If all healthy: return Healthy() with NO log (HLTH-07 -- silent 200)
    - HLTH-05/06/07: Staleness check with diagnostic log on failure, silent on success
    - Use `IReadOnlyDictionary<string, object>` for HealthCheckResult.Unhealthy data parameter (cast staleEntries appropriately)

    **Better approach for interval registration in AddScheduling:** Instead of trying to resolve IJobIntervalRegistry during registration, populate the registry by making JobIntervalRegistry directly available. Register `JobIntervalRegistry` as both concrete and interface:
    - In AddSimetraHealthChecks: `services.AddSingleton<JobIntervalRegistry>()` then `services.AddSingleton<IJobIntervalRegistry>(sp => sp.GetRequiredService<JobIntervalRegistry>())`
    - In AddScheduling: Create the `JobIntervalRegistry` instance inline, register it, and populate it during Quartz configuration:

    Actually, the cleanest approach: AddScheduling already has access to all interval values (heartbeatOptions.IntervalSeconds, correlationOptions.IntervalSeconds, poll.IntervalSeconds). Create the JobIntervalRegistry instance directly in AddScheduling, populate it with all intervals, and register it as a singleton:

    ```csharp
    var intervalRegistry = new JobIntervalRegistry();
    // After heartbeat trigger registration:
    intervalRegistry.Register("heartbeat", heartbeatOptions.IntervalSeconds);
    // After correlation trigger registration:
    intervalRegistry.Register("correlation", correlationOptions.IntervalSeconds);
    // After each state poll trigger:
    intervalRegistry.Register($"state-poll-{module.DeviceName}-{poll.MetricName}", poll.IntervalSeconds);
    // After each metric poll trigger:
    intervalRegistry.Register($"metric-poll-{device.Name}-{poll.MetricName}", poll.IntervalSeconds);
    services.AddSingleton<IJobIntervalRegistry>(intervalRegistry);
    ```

    Move IJobIntervalRegistry registration from AddSimetraHealthChecks into AddScheduling (where the interval data lives). AddSimetraHealthChecks just registers the three health checks.
