    **GracefulShutdownService.cs** (namespace Simetra.Lifecycle):
    - Sealed class implementing IHostedService (not BackgroundService -- no ExecuteAsync needed)
    - Constructor injects:
      - `ISchedulerFactory` (to put scheduler in standby)
      - `IDeviceChannelManager` (to drain channels)
      - `IServiceProvider` (to resolve MeterProvider/TracerProvider via GetService -- null-safe)
      - `ILogger<GracefulShutdownService>`
    - `StartAsync(CancellationToken ct)`: return Task.CompletedTask (no startup work)
    - `StopAsync(CancellationToken ct)`: orchestrates the shutdown sequence:

      ```
      // LIFE-05: Ordered shutdown steps.
      // Step 1 (Release lease) and Step 2 (Stop listener) are handled by their own
      // IHostedService.StopAsync in reverse registration order AFTER this service.
      // K8sLeaseElection and SnmpListenerService stop naturally in their own StopAsync.
      //
      // This service handles Steps 3-5:

      // Step 3: Put scheduler in standby (prevents new job fires)
      await ExecuteWithBudget("PauseScheduler", TimeSpan.FromSeconds(5), async () =>
      {
          var scheduler = await _schedulerFactory.GetScheduler();
          await scheduler.Standby();
          _logger.LogInformation("Scheduler placed in standby");
      }, ct);

      // Step 4: Drain device channels
      await ExecuteWithBudget("DrainChannels", TimeSpan.FromSeconds(10), async () =>
      {
          _channelManager.CompleteAll();
          await _channelManager.WaitForDrainAsync(CancellationToken.None);
          _logger.LogInformation("Device channels drained");
      }, ct);

      // Step 5: Flush telemetry (LIFE-07: protected with own budget regardless of prior outcomes)
      // This ALWAYS runs even if Steps 3-4 fail or are abandoned.
      await FlushTelemetryAsync();
      ```

    - `ExecuteWithBudget` private method:
      - Creates a linked CancellationTokenSource from the outer token with CancelAfter(budget)
      - Tries the action; catches OperationCanceledException when step CTS is cancelled (log Warning: step exceeded budget)
      - Catches generic Exception (log Error: step failed)
      - LIFE-06: Each step has bounded time budget; if exceeded, abandon and move to next

    - `FlushTelemetryAsync` private method (LIFE-07: protected with own budget):
      - This uses its OWN CancellationTokenSource(TimeSpan.FromSeconds(5)) -- NOT linked to the outer token
      - Resolves MeterProvider and TracerProvider via `_serviceProvider.GetService<T>()` (null-safe, matches existing pattern from Phase 7)
      - Calls ForceFlush(timeoutMilliseconds: 5000) on each
      - Wrapped in try/catch to ensure this step completes regardless
      - Log Information: "Telemetry flush completed"

    Time budget allocation (total ~30s):
    - Steps 1-2 (lease + listener): handled by framework reverse-order StopAsync (~6s)
    - Step 3 (scheduler standby): 5s budget
    - Step 4 (drain channels): 10s budget
    - Step 5 (telemetry flush): 5s budget (protected)
    - Reserve: ~4s for framework overhead
