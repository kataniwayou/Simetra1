**IProcessingCoordinator.cs:**
- Namespace: `Simetra.Pipeline`
- Single method: `void Process(ExtractionResult result, DeviceInfo device, string correlationId)`
- XML doc explaining: processes an extraction result through Branch A (metrics) and optionally Branch B (State Vector) based on source routing rules. Source=Module routes to both branches. Source=Configuration routes to Branch A only.

**ProcessingCoordinator.cs:**
- Namespace: `Simetra.Pipeline`
- Sealed class implementing IProcessingCoordinator
- Constructor injection:
  - `IMetricFactory metricFactory`
  - `IStateVectorService stateVector`
  - `ILogger<ProcessingCoordinator> logger`

**Process method implementation (this is the critical part -- PROC-06 + PROC-08):**

```
public void Process(ExtractionResult result, DeviceInfo device, string correlationId)
{
    // Branch A: Metrics -- ALWAYS runs (both Module and Configuration sources)
    try
    {
        _metricFactory.RecordMetrics(result, device);
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Branch A (metrics) failed for {MetricName} on {DeviceName}",
            result.Definition.MetricName, device.Name);
    }

    // Branch B: State Vector -- ONLY for Source=Module (PROC-06)
    if (result.Definition.Source == MetricPollSource.Module)
    {
        try
        {
            _stateVector.Update(device.Name, result.Definition.MetricName, result, correlationId);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Branch B (state vector) failed for {MetricName} on {DeviceName}",
                result.Definition.MetricName, device.Name);
        }
    }
}
```

**Key design decisions:**
- Branch A runs FIRST, Branch B runs SECOND (ordering is deterministic but they are independent)
- Each branch has its own try/catch -- a failure in Branch A does NOT prevent Branch B from running (PROC-08)
- Source check (`result.Definition.Source == MetricPollSource.Module`) is OUTSIDE Branch B's try/catch -- this is routing logic, not Branch B execution
- Method is synchronous (void, not Task) -- both branches are CPU-bound, in-memory operations (Pitfall 6 from research)
- PROC-02 (leader-only OTLP gating) is NOT implemented here -- gating happens at MeterProvider exporter level in Phase 7/8 (confirmed by research open question 1)

```csharp
/// <summary>
/// Registers all Phase 4 processing pipeline services: metric factory, state vector,
/// and processing coordinator. Must be called after <see cref="AddSimetraConfiguration"/>
/// (MetricFactory depends on IOptions&lt;SiteOptions&gt;).
/// </summary>
public static IServiceCollection AddProcessingPipeline(this IServiceCollection services)
{
    services.AddSingleton<IMetricFactory, MetricFactory>();
    services.AddSingleton<IStateVectorService, StateVectorService>();
    services.AddSingleton<IProcessingCoordinator, ProcessingCoordinator>();

    return services;
}
```

**All three are singletons because:**
- MetricFactory holds the Meter and instrument cache -- must be single instance
- StateVectorService holds the ConcurrentDictionary -- must be single instance (the shared state)
- ProcessingCoordinator is stateless but its dependencies are singletons
