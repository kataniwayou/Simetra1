Create the IMetricFactory interface and MetricFactory implementation for Branch A metric creation.

**IMetricFactory.cs:**
- Namespace: `Simetra.Pipeline`
- Single method: `void RecordMetrics(ExtractionResult result, DeviceInfo device)`
- XML doc explaining it creates metrics with enforced base labels

**MetricFactory.cs:**
- Namespace: `Simetra.Pipeline`
- Sealed class implementing IMetricFactory
- Constructor injection: `IMeterFactory meterFactory`, `IOptions<SiteOptions> siteOptions`, `ILogger<MetricFactory> logger`
- Create a `Meter` via `meterFactory.Create("Simetra.Metrics")` in constructor
- Store `_siteOptions = siteOptions.Value` in constructor (read once, not per-call)
- Instrument cache: `private readonly ConcurrentDictionary<string, object> _instruments = new()`

**RecordMetrics implementation:**
1. Iterate `result.Metrics` (each entry is PropertyName -> long value)
2. For each entry, construct metric name: `$"{result.Definition.MetricName}_{propertyName}"`
3. Build TagList with 4 base labels (PROC-03):
   - `"site"` -> `_siteOptions.Name`
   - `"device_name"` -> `device.Name`
   - `"device_ip"` -> `device.IpAddress`
   - `"device_type"` -> `device.DeviceType`
4. Add dynamic labels from `result.Labels` (PROC-04): iterate and add each to TagList
5. Get or create instrument via `GetOrCreateInstrument(metricName, result.Definition.MetricType)`
6. Record value using the instrument with tags

**GetOrCreateInstrument(string name, MetricType type) -> object:**
- Use `_instruments.GetOrAdd(name, n => type switch { ... })`
- MetricType.Gauge -> `_meter.CreateGauge<long>(n)`
- MetricType.Counter -> `_meter.CreateCounter<long>(n)`
- Default case -> throw `ArgumentOutOfRangeException(nameof(type))`

**RecordValue(object instrument, long value, TagList tags):**
- If instrument is `Gauge<long> gauge` -> `gauge.Record(value, tags)`
- If instrument is `Counter<long> counter` -> `counter.Add(value, tags)`
- Else throw `InvalidOperationException` (should never happen if GetOrCreate is correct)

**Error handling in RecordMetrics:** Wrap each metric recording iteration in try/catch. If one metric fails (e.g., instrument creation error), log at Error and continue to next metric. Do NOT let one bad metric kill the entire batch.

**Important anti-patterns to avoid:**
- Do NOT create instruments per measurement -- cache them (Pitfall 1 from research)
- Do NOT add correlationId as a metric tag (Pitfall 2 -- high cardinality)
- Do NOT import OpenTelemetry packages -- use only System.Diagnostics.Metrics (Pitfall 3)
- Do NOT use ObservableGauge -- use Gauge<T>.Record() which is push-based (.NET 9)
- Use TagList struct (not params KeyValuePair[]) since we always have 4+ tags (Pitfall 5)
- Keep method synchronous -- Gauge.Record() and Counter.Add() are synchronous (Pitfall 6)

**Usings needed:**
- `System.Collections.Concurrent`
- `System.Diagnostics.Metrics`
- `Microsoft.Extensions.Logging`
- `Microsoft.Extensions.Options`
- `Simetra.Configuration`
- `Simetra.Models`

**StateVectorEntry.cs:**
- Namespace: `Simetra.Pipeline`
- Sealed class with init-only properties (same pattern as ExtractionResult -- mutable construction, immutable consumption):
  - `public required ExtractionResult Result { get; init; }`
  - `public required DateTimeOffset Timestamp { get; init; }`
  - `public required string CorrelationId { get; init; }`
- XML doc explaining this is a single State Vector entry holding last-known domain data

**IStateVectorService.cs:**
- Namespace: `Simetra.Pipeline`
- Methods:
  - `void Update(string deviceName, string metricName, ExtractionResult result, string correlationId)` -- updates the state for a device/definition
  - `StateVectorEntry? GetEntry(string deviceName, string metricName)` -- returns the entry or null if not present
  - `IReadOnlyDictionary<string, StateVectorEntry> GetAllEntries()` -- returns snapshot of all entries (for health probes in Phase 9)
- XML doc explaining State Vector is in-memory, no persistence, no TTL (PROC-07)

**StateVectorService.cs:**
- Namespace: `Simetra.Pipeline`
- Sealed class implementing IStateVectorService
- Constructor injection: `ILogger<StateVectorService> logger`
- Backing store: `private readonly ConcurrentDictionary<string, StateVectorEntry> _entries = new()`
- Composite key: `$"{deviceName}:{metricName}"` (Pitfall 4 from research -- must be device+definition, not just device)

**Update implementation:**
- Build composite key: `$"{deviceName}:{metricName}"`
- Use `_entries.AddOrUpdate(key, addValue, updateValueFactory)` where both add and update create a new StateVectorEntry with:
  - `Result = result`
  - `Timestamp = DateTimeOffset.UtcNow`
  - `CorrelationId = correlationId`
- Log at Debug level: "State Vector updated for {DeviceName}:{MetricName} with correlationId {CorrelationId}"

**GetEntry implementation:**
- Build composite key, use `_entries.TryGetValue`, return entry or null

**GetAllEntries implementation:**
- Return `new ReadOnlyDictionary<string, StateVectorEntry>(_entries.ToDictionary(kv => kv.Key, kv => kv.Value))`
- This creates a snapshot (not a live view) which is correct for health probe reads

**Important:**
- No persistence (PROC-07) -- no file writes, no serialization, no database
- No TTL -- entries live until overwritten or process restarts
- Thread safety via ConcurrentDictionary -- no manual locking needed

**Usings needed:**
- `System.Collections.Concurrent`
- `System.Collections.ObjectModel`
- `Microsoft.Extensions.Logging`
- `Simetra.Models`
