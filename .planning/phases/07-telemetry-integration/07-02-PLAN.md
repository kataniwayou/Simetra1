    **Add ForceFlush registration in Program.cs** after `var app = builder.Build();` and the existing correlationId setup, but before `app.Run()`.

    ```csharp
    // TELEM-05 / Success Criterion 5: ForceFlush during shutdown prevents final metric loss
    app.Lifetime.ApplicationStopping.Register(() =>
    {
        // Resolve providers if available (null-safe -- providers may not be registered in test scenarios)
        var meterProvider = app.Services.GetService<MeterProvider>();
        var tracerProvider = app.Services.GetService<TracerProvider>();

        // ForceFlush with 5-second timeout budget per provider
        meterProvider?.ForceFlush(timeoutMilliseconds: 5000);
        tracerProvider?.ForceFlush(timeoutMilliseconds: 5000);
        // LoggerProvider flush is handled by the host's logging infrastructure disposal
    });
    ```

    **Required usings** in Program.cs:
    - `using OpenTelemetry.Metrics;` (for `MeterProvider`)
    - `using OpenTelemetry.Trace;` (for `TracerProvider`)

    **Note:** `MeterProvider` and `TracerProvider` are resolved via `GetService` (not `GetRequiredService`) because they're null-safe -- the code should not throw if providers are absent in some test configuration. The `ForceFlush` call uses the null-conditional operator `?.` for safety.

    **Note:** This registers on `ApplicationStopping` which fires BEFORE the host disposes DI services. The OpenTelemetry hosting extension's own disposal also calls Shutdown() on providers, but explicit ForceFlush with a known time budget ensures queued batches are flushed before the SIGTERM grace period expires. Phase 9 will integrate this into the full time-budgeted shutdown sequence.
